## Project

**Zynthian** — open-source synthesizer platform running on Raspberry Pi. Three repositories cover system scripts, the Python UI/engine, and the web configuration tool.

- System scripts & config: `~/zynth/zynthian-sys/`
- UI & synth engine (Python): `~/zynth/zynthian-ui/`
- Web configuration interface: `~/zynth/zynthian-webconf/`
- Documentation: `~/zynth-docs/`
- Target hardware: Raspberry Pi 4 (also 3/5), ZynthianOS SD image

---

## Physical Layout

```
~/zynth/                       ← project root (NOT a git repo)
    zynthian-sys/              ← git repo: system scripts, config, boot, sbin
    zynthian-ui/               ← git repo: Python UI + synth engine
    zynthian-webconf/          ← git repo: web config interface (Python/Tornado)
    MD/   →  symlink           ← ~/zynth-docs/MD/
    CLAUDE.md  →  symlink      ← ~/zynth-docs/MD/CLAUDE.md

~/zynth-docs/                  ← git repo: documentation
    MD/                        ← session tracking files (NOT documentation)
    htmldoku/                  ← documentation SOURCE (.md files, hand-edited)
    docs/zynthian-Doku/        ← rendered HTML — generated by generate-html.py
```

**"Update documentation" means: edit `htmldoku/*.md`, run `generate-html.py`, commit both.**

Doc workflow:
```bash
cd ~/zynth-docs
# 1. edit htmldoku/somefile.md
python3 htmldoku/generate-html.py          # regenerates docs/zynthian-Doku/*.html
git add htmldoku/somefile.md docs/zynthian-Doku/somefile.html
git commit -m "docs: ..."
```

## Coding Rules (Python)

- PEP 8 style · 120-char line limit · no `print()` debug output (use `logging`)
- Synth engines inherit from `zynthian_engine` base class
- GUI screens inherit from `zynthian_gui_base`
- Config changes go through webconf handlers, not direct file edits
- Always check `zynthian_envars.sh` for environment variable names before hardcoding paths

## Session Workflow

### 1 — Orient

Read CLAUDE.md (auto) + `MD/inwork.md`. Both must be read before proceeding.

Present in-work items grouped by readiness:
- `[~]` partial — active development
- `[t]` in testing — re-verify on Pi before continuing
- `[>]` needs PR — submit before starting new work

Ask user to select one item before proceeding.

### 2 — Discover

Read the relevant `htmldoku/` page from the Documentation Lookup table. If wiki context needed, fetch from `wiki.zynthian.org`. Grep the relevant `zyngui/` or `zyngine/` file for existing implementations.

### 3 — Consult

Clarify:
- Which Pi model and hardware variant?
- Which ZynthianOS version (check `vcgencmd version` or webconf dashboard)?
- Is a Pi connected for live testing (`ssh root@zynthian.local`)?

End with a brief **sign-off card**:
```
Task:           <what we're doing>
Files affected: <list>
Test method:    SSH verify | webconf check | both
Proceed? ✓ / ✗
```

### 4 — Implement

One task per session. Log adjacent bugs in `MD/bugs.md` and finish current task first.

### 5 — Test

```bash
# Restart changed service on Pi
ssh root@zynthian.local "systemctl restart zynthian"
ssh root@zynthian.local "systemctl restart zynthian-webconf"

# Check status
ssh root@zynthian.local "systemctl status zynthian --no-pager"
ssh root@zynthian.local "journalctl -u zynthian -n 30 --no-pager"

# Check JACK
ssh root@zynthian.local "systemctl status jack2 --no-pager"
```

Manual browser verify: open `http://zynthian.local` and confirm behavior.

### 6 — Wrap up

- Move item from `MD/inwork.md` to `MD/done.md`
- Update `MD/bugs.md` / `MD/decisions.md` as needed
- Ask: **"Does the documentation need updating?"** → edit `htmldoku/*.md`, run `generate-html.py`, commit both

Commands reference: `MD/commands.md`

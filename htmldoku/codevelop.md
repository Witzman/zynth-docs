# Working with Claude on Zynthian Tutorials

This page describes how tutorials are created collaboratively with Claude Code. The approach: start with the smallest proof of concept, verify it works on the Pi, then extend step by step. Everything produced gets published to this documentation site.

---

## Setup

### Hardware

| What | Detail |
|------|--------|
| Hardware | Raspberry Pi 4, ZynthianOS |
| Physical controls | None — no hardware kit, no touchscreen |
| Access | SSH · webconf (`http://zynthian.local`) · VNC |

All tutorial steps are verified via SSH, webconf, or VNC. No step requires physical encoders, buttons, or a touchscreen.

### File Layout

```
~/zynth/                       ← project root (NOT a git repo)
    zynthian-sys/              ← system scripts, config, hardware detection
    zynthian-ui/               ← Python UI + synth engines
    zynthian-webconf/          ← web configuration interface
    MD/   →  symlink           ← ~/zynth-docs/MD/
    CLAUDE.md  →  symlink      ← ~/zynth-docs/MD/CLAUDE.md

~/zynth-docs/                  ← git repo: this documentation
    MD/                        ← session context files (not documentation)
    htmldoku/                  ← documentation source (.md files, hand-edited)
    docs/zynthian-Doku/        ← rendered HTML — generated by generate-html.py
```

The symlinks make `CLAUDE.md` and `MD/` visible from `~/zynth/` on every session. Claude Code reads `CLAUDE.md` automatically when opened from `~/zynth/`.

### Context Files

Claude reads three files in this order at the start of every session:

| # | File | What it contains |
|---|------|-----------------|
| 1 | `MD/CLAUDE.md` | Project identity, file locations, hardware constraints, doc lookup table |
| 2 | `MD/inwork.md` | Active tutorials with status — `[~]` drafting · `[t]` testing · `[>]` ready to publish |
| 3 | `MD/agent-behavior.md` | All behavioral rules — session protocol, tutorial creation process, writing rules, format rules |

`MD/agent-behavior.md` governs all agent behavior. Nothing happens until all three are read.

---

## Session Workflow

### 1 — Orient

Claude reads all three context files, presents active tutorials grouped by status and the future candidates list, then asks:

> **"What do you want to work on?"**

Nothing happens until you answer.

### 2 — New topic: Discovery

For a new tutorial topic, Claude runs an interactive discovery phase before drafting anything:

- **Feasibility check** — possible with SSH / webconf / VNC only? If not, says so immediately with an alternative.
- **Difficulty** — `Beginner` (webconf only) · `Intermediate` (SSH + Zynthian concepts) · `Advanced` (internals, config files, multiple systems)
- **Clarifying questions** — back and forth until scope is agreed

Claude then presents a roadmap for confirmation:

```
Goal:         <what we're building toward>
Difficulty:   Beginner | Intermediate | Advanced

PoC (Part 1): <smallest verifiable outcome>
Part 2:       <first extension>
Part 3:       <next extension>
...
Proceed?
```

No drafting starts until you confirm the roadmap.

### 3 — Research

Claude reads the relevant `htmldoku/` reference page(s) before writing any step. Documentation first — code only if the documentation has a gap.

### 4 — Draft

Claude drafts the full tutorial upfront. Each Part is tagged `[draft]`. Part 1 is presented first — you test it before Part 2 is introduced.

### 5 — Test on Pi

Follow each step and report back:
- `ok` → next step
- `failed: <what happened>` → Claude diagnoses, fixes, re-presents

When all steps in a Part pass, the tag updates to `[verified]` and the file is committed. Then Part 2 is introduced.

### 6 — Publish

When all Parts are `[verified]`:

```bash
cd ~/zynth-docs
# Add sidebar entry to generate-html.py if new page
python3 htmldoku/generate-html.py
git add htmldoku/project-<slug>.md docs/zynthian-Doku/project-<slug>.html
git commit -m "docs: add tutorial — <title>"
git push
```

Move the item from `MD/inwork.md` to `MD/done.md` as `[x]`.

## Tutorial Structure

Every tutorial follows this structure:

```markdown
# <Title>

**Goal:** One sentence — what you achieve by the end.
**Prerequisites:** What must already be working.
**Access:** SSH · Webconf · VNC (only what the tutorial uses)

---

## Part 1 — <PoC title>

One sentence intro.

### Step 1 — <action verb + object>

<Exact command or UI path>

```bash
# command
```

**Verify:** <what you should see or hear>

### Step 2 — ...

---

## Part 2 — <extension title>

...

---

## Going Further

Ideas for extending the setup once Part 1 works.
```

Tutorials live in `htmldoku/project-<slug>.md` and appear under **Personal Projects** in the sidebar.

## Documentation Generation

```bash
cd ~/zynth-docs
python3 htmldoku/generate-html.py
# → generates docs/zynthian-Doku/*.html
# → rebuilds sidebar navigation
# → rebuilds search-index.json
```

Preview locally: open `docs/zynthian-Doku/index.html` in a browser.

To add a new tutorial page:

1. Create `htmldoku/project-<slug>.md`
2. Add an entry under `Personal Projects` in the `SIDEBAR` list in `generate-html.py`
3. Run the generator
4. Commit source `.md` and rendered `.html` together

## Confidence Markers

Documentation pages can flag uncertain content with `[low]`:

```markdown
The reverb plugin requires at least 256MB free RAM. [low]
```

The generator renders `[low]` in a muted style. Use it for:
- Information derived from the wiki but not verified hands-on
- Version-dependent behavior
- Observations from a single hardware configuration
